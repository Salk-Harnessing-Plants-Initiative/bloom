# Bloom Reproducible Setup ‚Äî Design Spec

## Goal

Make Bloom deployable from a fresh `git clone` with `make init && make dev-up`. Target audience: external users and open-source contributors with no prior Bloom knowledge.

## Architecture Decision

**Hand-maintained compose file + automated init script for secrets and volumes.**

- Keep `docker-compose.dev.yml` as a single, hand-maintained file with all services (Supabase + Bloom)
- Add `make init` to automate secret generation, Supabase volumes download, and prerequisite checks
- Add `make upgrade-supabase` to compare image versions and env vars between releases
- Each environment (dev, staging, prod) is a separate directory with its own repo clone, secrets, and data

### Why hand-maintained?

We evaluated and rejected several "cleaner" approaches:

| Approach | Why rejected |
|---|---|
| Docker Compose `include` | Can't override included service properties |
| Docker Compose `-f` merge | `--project-directory` can't resolve paths for both files |
| Git submodule for Supabase | Repo is 2.3 GB; we need ~1 MB |
| Python-generated compose | Over-engineering: YAML fidelity issues, ambiguous source of truth, harder to debug |
| Template-based compose | Fragile string interpolation |

### Why separate directories per environment?

We initially designed staging as port offsets on the same compose file (`-p bloom-staging` with env var ports). Reviews found this approach has multiple problems:
- Hardcoded `container_name` directives ignore `-p` flag (collision)
- Every host-exposed port needs parameterization (~13 ports)
- Shared bind mounts for storage, bloommcp data
- Complex, error-prone, hard to reason about

The separate-directory model is simpler and safer:
- Each environment is a full repo clone in its own directory
- Each has its own `.env`, data directories, Docker project
- Impossible for staging to touch production data
- No port parameterization needed ‚Äî each environment owns its ports
- Standard Docker Compose with zero special flags

## Environment Model

```
Local laptop:
  ~/repos/salk-bloom/              ‚Üí dev (make dev-up)

bloom-dev server:
  /data/bloom-staging/             ‚Üí staging (make dev-up)
  /data/bloom/                     ‚Üí production (make prod-up)
```

### Isolation guarantees

| Property | Dev (laptop) | Staging (server) | Production (server) |
|---|---|---|---|
| Directory | `~/repos/salk-bloom/` | `/data/bloom-staging/` | `/data/bloom/` |
| Compose file | `docker-compose.dev.yml` | `docker-compose.dev.yml` | `docker-compose.prod.yml` |
| Env file | `.env.dev` | `.env.dev` | `.env.prod` |
| Postgres data | `./volumes/db/data` | `./volumes/db/data` | `./volumes/db/data` |
| MinIO data | `./minio_data` | `./minio_data` | configured path |
| JWT_SECRET | unique | unique | unique |
| POSTGRES_PASSWORD | unique | unique | unique |
| Ports | standard | standard | standard |

Each directory is self-contained. `./volumes/db/data` is a relative path ‚Äî it resolves to a different absolute path in each directory.

### Upgrade workflow with staging

1. On bloom-dev, `cd /data/bloom-staging`
2. `git pull` to get updated compose + image tags
3. `make upgrade-supabase` ‚Äî check what changed
4. Apply changes, `make init --download-only`
5. `make dev-up` ‚Äî start staging with new versions
6. `make check` ‚Äî verify health
7. If good: `cd /data/bloom && git pull` and repeat for production

## What Changes in `docker-compose.dev.yml`

### 1. Remove all `container_name` directives

Remove from all 11 services: `supabase-studio-dev`, `supabase-kong`, `realtime-dev.supabase-realtime-dev`, `supabase-minio`, `supabase-storage`, `supabase-imgproxy-dev`, `supabase-meta-dev`, `supabase-analytics-dev`, `db-dev`, `swagger-ui-dev`, `supabase-pooler`.

Docker Compose auto-generates names with the project prefix. This prevents name collisions if multiple stacks run on the same host (staging + production on bloom-dev).

**Migration note:** `docker exec db-dev ...` becomes `docker compose exec db-dev ...` (uses service name, not container name). Update all Makefile targets that use hardcoded container names:
- `reset-storage` (greps/execs `db-dev`, `supabase-minio`)
- `apply-migrations-local` (greps `db-dev`)
- `load-test-data` (greps `db-dev`)
- `upload-images` (greps `supabase-minio`)
- `gen-types` (greps `db-dev`)
- `list-buckets` (greps `supabase-minio`)

All should switch to `docker compose -f docker-compose.dev.yml exec <service-name> ...`

This divergence is documented with a comment in the compose file explaining why and what would need to change to align with a future Supabase release.

### 3. Add missing health checks

| Service | Health check | Notes |
|---|---|---|
| bloom-web | `node -e "fetch('http://localhost:3000').then(r=>{if(!r.ok)process.exit(1)})"` | No `curl` in node:20-bookworm; Node 18+ has built-in fetch |
| supabase-minio | `curl -sf http://localhost:9000/minio/health/live` | MinIO images include curl |
| kong | `kong health` | Built-in Kong CLI command |
| meta | `wget --no-verbose --tries=1 --spider http://localhost:8080/health` | node-alpine has `wget`, not `curl` |

Skip: PostgREST (`rest`) ‚Äî minimal binary, no curl/wget. swagger-ui ‚Äî nginx-based, no curl/wget, non-critical.

### 4. Add missing env vars to compose

| Variable | Service | Value |
|---|---|---|
| `LANGCHAIN_POSTGRES_URL` | langchain-agent | `postgresql://postgres:${POSTGRES_PASSWORD}@db-dev:5432/postgres` |
| `MINIO_STORAGE_S3_ENDPOINT` | storage | `http://supabase-minio:9000` |
| `MINIO_S3_STORAGE_REGION` | storage | `us-east-1` |
| `POOLER_DB_POOL_SIZE` | supavisor | `${POOLER_DB_POOL_SIZE:-5}` |

### 5. Make `CORS_ORIGINS` use env var

Current compose hardcodes `CORS_ORIGINS: "http://localhost:3000"`. Change to `CORS_ORIGINS: ${CORS_ORIGINS:-http://localhost:3000}` so production can override it.

### 6. Fix `OPENAI_API_KEY` empty-string handling

Current: `OPENAI_API_KEY: ${OPENAI_API_KEY}` ‚Äî expands to empty string when not set, which causes OpenAI SDK auth errors.

Fix: Remove `OPENAI_API_KEY` from the compose `environment:` block. Instead, pass it via the `.env` file mechanism only when set. The langchain agent code should treat empty string same as unset.

## Init Script: `scripts/init_dev.py`

Run via: `uv run --with pyjwt,python-dotenv scripts/init_dev.py [--force] [--download-only]`

### Flow

1. **Check prerequisites**
2. **Check idempotency** (before any work)
3. **Download Supabase volumes** (if needed)
4. **Generate `.env.dev`**
5. **Conditional vLLM check**
6. **Print summary**

### Step 1: Check prerequisites

- Docker installed, `docker compose version` works
- Docker Compose >= 2.20
- `uv` installed ‚Äî if not: print `curl -LsSf https://astral.sh/uv/install.sh | sh`
- Fail early with clear messages

### Step 2: Check idempotency

- If `.env.dev` exists and `--force` not set:
  - Print warning: "`.env.dev` already exists. Re-generating will invalidate all secrets."
  - Print: "Use `--force` to overwrite (backs up to `.env.dev.backup.<timestamp>`)"
  - Exit non-zero
- If `--force`: will backup existing file in Step 4

### Step 3: Download Supabase volumes

- Read version from `SUPABASE_VERSION` file
- If `volumes/.supabase-version` exists and matches: skip download
- If exists but doesn't match: warn and re-download
- Download `https://github.com/supabase/supabase/archive/refs/tags/{version}.tar.gz`
- Extract using Python `tarfile` module:
  - `docker/volumes/` ‚Üí `./volumes/` (strip 2 path components: `supabase-X.XX.XX/docker/`)
  - `docker/.env.example` ‚Üí temp file (for parsing upstream defaults)
- `mkdir -p volumes/` before extraction
- Write `volumes/.supabase-version` marker
- Verify expected files exist (kong.yml, *.sql, pooler.exs)
- If download fails: clean up partial state, print clear error with "Network required for first run"
- If `--download-only`: stop here

### Step 4: Generate `.env.dev`

- Parse extracted upstream `.env.example` using `python-dotenv` (all Supabase vars with defaults)
- Parse `.env.example` (Bloom-specific vars)
- If `--force` and `.env.dev` exists: backup to `.env.dev.backup.<timestamp>`
- Generate secrets:

| Variable | Method | Notes |
|---|---|---|
| POSTGRES_PASSWORD | `secrets.token_urlsafe(24)` | ~32 chars |
| JWT_SECRET | `secrets.token_urlsafe(48)` | ~64 chars |
| ANON_KEY | JWT: role=anon, iss=supabase, iat=now, exp=now+10yr, HS256 | |
| SERVICE_ROLE_KEY | JWT: role=service_role, same params | |
| VAULT_ENC_KEY | `secrets.token_hex(16)` | Exactly 32 hex chars |
| SUPAVISOR_ENC_KEY | `secrets.token_hex(32)` | 64 hex chars |
| SECRET_KEY_BASE | `secrets.token_urlsafe(48)` | ~64 chars |
| DB_ENC_KEY | `secrets.token_urlsafe(24)` | Used by Realtime |
| DASHBOARD_PASSWORD | `secrets.token_urlsafe(16)` | |
| PG_META_CRYPTO_KEY | `secrets.token_urlsafe(24)` | Required by meta and studio |
| LOGFLARE_PUBLIC_ACCESS_TOKEN | `secrets.token_urlsafe(24)` | |
| LOGFLARE_PRIVATE_ACCESS_TOKEN | `secrets.token_urlsafe(24)` | |
| MINIO_ROOT_PASSWORD | `secrets.token_urlsafe(16)` | |
| BLOOMMCP_API_KEY | `secrets.token_urlsafe(24)` | |

Note: `PG_META_CRYPTO_KEY` is not referenced in the current compose but is in the upstream `.env.example` and may be needed for Studio features. Generate it to be safe.

- Copy all upstream defaults from parsed `.env.example`
- Set Bloom-specific overrides:
  - `SITE_URL=http://localhost:3000`
  - `STUDIO_DEFAULT_ORGANIZATION=Bloom`
  - `STUDIO_DEFAULT_PROJECT=Bloom`
  - `POSTGRES_HOST=db-dev`
  - `POOLER_TENANT_ID=bloom-dev`
  - `S3_PROTOCOL_ACCESS_KEY_ID` = MINIO_ROOT_USER value
  - `S3_PROTOCOL_ACCESS_KEY_SECRET` = MINIO_ROOT_PASSWORD value
  - `DASHBOARD_USERNAME=supabase`
  - `LANGCHAIN_POSTGRES_URL=postgresql://postgres:{POSTGRES_PASSWORD}@db-dev:5432/postgres`
  - `MINIO_STORAGE_S3_ENDPOINT=http://supabase-minio:9000`
  - `MINIO_S3_STORAGE_REGION=us-east-1`
  - `POOLER_DB_POOL_SIZE=5`
- Prompt for optional: `LOCAL_LLM_URL`, `MINIO_DATA_PATH`
- Write `.env.dev`
- Set file permissions to 600 on Linux/macOS

### Step 5: Conditional vLLM check

- If `LOCAL_LLM_URL` was set, `GET {url}/models` with 5-second timeout
- Success: print model name
- Failure: print warning (not blocking)

### Step 6: Summary

```
Setup complete!
  make dev-up     Start all services
  make check      Verify health
  make dev-logs   View logs
```

## Makefile Targets

```makefile
SHELL := bash
COMPOSE_DEV := docker compose -f docker-compose.dev.yml --env-file .env.dev

.PHONY: init dev-up dev-down dev-logs check download-volumes upgrade-supabase

init:                              ## First-time setup
	uv run --with pyjwt,python-dotenv scripts/init_dev.py

dev-up:                            ## Start all services
	$(COMPOSE_DEV) up -d --build

dev-down:                          ## Stop all services
	$(COMPOSE_DEV) down

dev-logs:                          ## Tail all logs
	$(COMPOSE_DEV) logs -f

check:                             ## Verify all services healthy
	uv run scripts/check_health.py

download-volumes:                  ## Re-download Supabase volumes (after updating SUPABASE_VERSION)
	uv run --with pyjwt,python-dotenv scripts/init_dev.py --download-only

upgrade-supabase:                  ## Compare current vs new Supabase release
	@echo "Current pinned version: $$(cat SUPABASE_VERSION)"
	@read -p "New version to compare (e.g., v1.27.00): " new_ver; \
	old_ver=$$(cat SUPABASE_VERSION); \
	old_compose=$$(curl -sf "https://raw.githubusercontent.com/supabase/supabase/$$old_ver/docker/docker-compose.yml"); \
	new_compose=$$(curl -sf "https://raw.githubusercontent.com/supabase/supabase/$$new_ver/docker/docker-compose.yml"); \
	if [ -z "$$old_compose" ] || [ -z "$$new_compose" ]; then \
	  echo "ERROR: Failed to download compose files. Check version tags."; exit 1; \
	fi; \
	echo ""; \
	echo "=== Image version changes ($$old_ver ‚Üí $$new_ver) ==="; \
	diff <(echo "$$old_compose" | grep 'image:' | sort) <(echo "$$new_compose" | grep 'image:' | sort) || true; \
	echo ""; \
	echo "=== New env vars in $$new_ver ==="; \
	old_vars=$$(curl -sf "https://raw.githubusercontent.com/supabase/supabase/$$old_ver/docker/.env.example"); \
	new_vars=$$(curl -sf "https://raw.githubusercontent.com/supabase/supabase/$$new_ver/docker/.env.example"); \
	comm -13 <(echo "$$old_vars" | grep -v '^#' | grep '=' | cut -d= -f1 | sort) \
	         <(echo "$$new_vars" | grep -v '^#' | grep '=' | cut -d= -f1 | sort); \
	echo ""; \
	echo "Next steps:"; \
	echo "  1. Update image tags in docker-compose.dev.yml"; \
	echo "  2. Add any new env vars to .env.example and scripts/init_dev.py"; \
	echo "  3. Update SUPABASE_VERSION to $$new_ver"; \
	echo "  4. Run: make download-volumes"; \
	echo "  5. Test: make dev-up && make check"
```

Note: `SHELL := bash` requires `bash` on PATH. Works in Git Bash (run Make from Git Bash terminal), Linux, and macOS.

## Upgrade Workflow

1. `make upgrade-supabase` ‚Äî shows image changes and new env vars between versions
2. Update image tags in `docker-compose.dev.yml`
3. Add any new env vars to `.env.example` and `scripts/init_dev.py`
4. Update `SUPABASE_VERSION` file
5. `make download-volumes` ‚Äî download matching init scripts
6. On staging server (`/data/bloom-staging/`): `git pull && make dev-up && make check`
7. If staging is healthy: on production (`/data/bloom/`): `git pull && make prod-up`
8. Commit all changes
9. **Rollback:** revert commit, `make download-volumes`, restart

## Migration Guide

### For local developers

```
1. Stop old containers:
   docker compose -f docker-compose.dev.yml down

2. container_name directives have been removed. Update commands:
   Old: docker exec db-dev psql ...
   New: docker compose -f docker-compose.dev.yml exec db-dev psql ...

3. Your database data at ./volumes/db/data is preserved.

4. Run init to download new volumes and generate secrets:
   make init
   # If you have existing .env.dev with data you want to keep,
   # do NOT use --force. Manually add new variables from .env.example.

5. Start and verify:
   make dev-up
   make check
```

### For bloom-dev server

```
1. Stop containers:
   cd /data/bloom && docker compose -f docker-compose.dev.yml down

2. Kong stays on 3.9.1 (no revert needed).

3. Re-download volumes from pinned version:
   make download-volumes

4. Review .env ‚Äî add new variables from .env.example.
   Do NOT regenerate secrets if preserving existing data.

5. Restart:
   make dev-up && make check

6. Set up staging directory:
   cd /data && git clone <repo> bloom-staging
   cd bloom-staging && make init
   make dev-up && make check
```

## Decisions Log

| Decision | Chosen | Rejected | Why |
|---|---|---|---|
| Compose strategy | Hand-maintained single file | `include`, `-f` merge, generated, template | Simplest, most debuggable, best onboarding |
| Supabase volumes | Download at init | Git submodule, committed | Third-party out of git; submodule is 2.3 GB |
| Secret management | Auto-generate at init | Hardcoded dev secrets | Unique per install, secure |
| Init script language | Python via `uv` | Bash | Cross-platform, JWT needs pyjwt |
| Environment model | Separate directories | Port offsets on same compose | Simpler, safer, no port parameterization |
| container_name | Remove all | Keep | Required for multi-stack on same host |
| Kong version | Stay on 3.9.1 | Revert to 2.8.1 | Already working; documented divergence |
| VAULT_ENC_KEY | `token_hex(16)` = 32 hex chars | `token_urlsafe(32)` = 43 chars | Supavisor requires exactly 32 chars |
| Makefile SHELL | `SHELL := bash` | `/bin/bash` | PATH-based; works across platforms |
| Volumes version check | Marker file `.supabase-version` | No check | Prevents stale volumes |
| Upgrade tool | Image tag + env var comparison | Raw file diff | Raw diff is noise due to structural differences |
