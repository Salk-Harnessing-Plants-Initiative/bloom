# bloom-mcp Phase 2 ‚Äî vertical-slice roadmap

**Status: APPROVED (Elizabeth, 2026-06-15) ‚Äî adversarially reviewed; tracking issues filed (#305‚Äì#309). Implement tier-by-tier via `/new-feature`.**

**Design (source of truth):** vault `docs/superpowers/specs/2026-06-15-bloom-mcp-phase2-design.md` + **`2026-06-15-bloom-mcp-phase2-persistence-design.md`** (persistence/provenance ‚Äî corrects the IO layer to two ports over the deployed Supabase adapter). Copies in this repo at `bloommcp/docs/`. Master spec: `2026-05-11-metcalf-2026-evelyn-bloom-mcp-design.md`.

**Program:** harden the `bloommcp` prototype into a thin, validated MCP server that **delegates all analysis to `sleap-roots-analyze`**, proven via a **depth vertical slice** (contract pattern end-to-end on 1‚Äì2 fast tools). One **OpenSpec change + PR per tier**, TDD, tier-by-tier per the `roadmap-driven-pipeline` workflow.

**How each tier is built:** run the repo's **`/new-feature`** command (staging-first): branch off `origin/staging` ‚Üí `/openspec:proposal` (`tasks.md` TDD-ordered, **oracle test first**) ‚Üí `/review-openspec` (reconcile BLOCKING/IMPORTANT literally) ‚Üí **Elizabeth approval** ‚Üí `/openspec:apply` (TDD) ‚Üí `/pre-merge` ‚Üí PR **to `staging`** (link the tier issue + this roadmap) ‚Üí `/openspec:archive`. Each tier issue (#305‚Äì#309) carries these steps.

**Oracle, generally:** bloom-mcp tools are validated by **reproducing `sleap-roots-analyze`'s golden values *through* the MCP tool** (the #120 wheat-EDPIE fixtures) + the contract's 5 test patterns. Tracking: **one issue per tier** (in `Salk-Harnessing-Plants-Initiative/bloom`).

## Live-state facts (verified 2026-06-15)
- **The delegated functions are ALREADY public** in `sleap-roots-analyze` **v0.1.0a2**: `perform_pca_analysis`, `perform_kmeans_clustering`, `perform_gmm_clustering` are in `__all__`. So Tiers 3‚Äì4 are **not** blocked on a new release.
- **The result *types* are in-flight, additive adapters:** `PCAResult` (#149), `HeritabilityResult` (#150), `ClusterResult`/`KMeansResult`/`GMMResult` (#151) are **open PRs** (a reviewed, ready stack) over the dict the `perform_*` functions already return. Consuming them is a **preferred upgrade** (avoids duplicating a ~30-line adapter), **not** a hard gate.
- `sleap-roots-contracts 0.1.0a1` **is** on PyPI (exports `validate_analysis_input` + `canonicalize_role_dtypes`). analyze wires contracts via a PyPI pin in **open PR #155** (not a git ref; not merged).
- **bloommcp today has NO `sleap-roots-analyze` or `sleap-roots-contracts` dependency** (the `source/` files are self-contained vendored sklearn). Tier 0 **introduces** them.
- **bloommcp already reads from AND writes to Supabase Storage as `bloom_agent`** (single `BLOOM_AGENT_KEY`): reads `bloommcp_input/` CSVs (`read_input_csv`) + tables via PostgREST; writes versioned outputs via `AnalysisWriter` ‚Üí `bloommcp_output/.../v<N>/` + `manifest.json` (RLS migration `20260605000000`). The slice **builds on this behind two ports** (read + write) ‚Äî it is NOT read-only and NOT CSV-on-disk.
- salk-bloom is **staging-first**: feature work branches off `origin/staging` and PRs target **`staging`**; `main` is the deploy/prod branch (only promotion PRs target it). The bloom agent is **not live**.
- **Benfica reviewed + APPROVED PR #310 (2026-06-15)** and confirmed the legacy `run_X_workflow` tools + `source/*` can be **retired once Stage 1 (Tiers 0‚Äì4) lands** (so retirement is *after Stage 1*, not indefinitely deferred), and that `code_versions` should include `sleap-roots-analyze` + `sleap-roots-contracts` versions.

## Dev setup (ready)
`DEV_SETUP.md` + `make init` (mints JWT-signed `BLOOM_AGENT_KEY`) bring up the full stack via `docker-compose.dev.yml` ‚Äî **bloommcp + langchain-agent + self-hosted Supabase + MinIO + storage-api**. The dev `/health` healthcheck is fixed (probes the **unauthenticated** `/health` custom route in `server.py`; **#125 resolved** by #281 on staging). Two Tier-0/2 preconditions surfaced from the deployed code:
- **Windows: develop inside WSL2 on the native Linux FS** (`~/repos/bloom`), **NOT `/mnt/c/...`** (CRLF/filesystem issues ‚Äî #124). Onboarding note for Evelyn.
- **`supabase_client.py` raises at import** if `SUPABASE_URL`/`BLOOM_AGENT_KEY` are unset ‚Üí Tier 0 must make that validation **lazy** so `import bloom_mcp` + fakes-based unit tests run with **no Supabase**.

## Hard constraints / decisions (design)
Thin delegation (re-orchestrate nothing) ¬∑ small flat fast surface (Qwen3.5-9B + Claude) ¬∑ flat `src/bloom_mcp/tools/` (no `v1/`, no `generated/manual`) ¬∑ package-SemVer + stable names (no `_v2`) ¬∑ **two persistence ports** (`ExperimentReader` read + `ResultStore` write) over the deployed Supabase adapter ¬∑ **provenance unified with the manifest `VersionEntry`** (schema v2‚Üív3 additive: +`seed`/`agent`/`output_sha256`) ¬∑ **return links not blobs** ¬∑ auth/transport unchanged. **Deferred (design ¬ß8 + persistence design ¬ß8):** URL-namespace, `find_tools`/RAG, async-pipeline tools, api-diff gate, Phase-3 generator, retirement of `source/*` + `run_X_workflow` (**after Stage 1 / Tiers 0‚Äì4 ‚Äî Benfica confirmed on PR #310**), **DB-direct read adapter (sub-project #2), orchestrator-owned/per-user-identity writes + real RLS (Benfica), manifest compare-and-swap (multi-instance)**.
